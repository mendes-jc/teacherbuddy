# TeacherBuddy Phase 3 (Document RAG) Implementation Plan

> **For agentic workers:** implement task-by-task.

**Goal:** Let teachers add their own teaching material and have AI suggestions grounded in it — retrieval-augmented generation via pgvector, with a decoupled embeddings layer.

**Architecture:** Documents are chunked and embedded (provider-agnostic `EmbeddingProvider`, default Gemini free tier) into a pgvector column. At suggestion time we embed a query built from the student's situation, retrieve the top-k relevant chunks via a Postgres similarity function, and pass them into the existing provider-neutral suggestion context. RAG degrades gracefully: if embeddings aren't configured or retrieval fails, suggestions still generate without references.

**Tech Stack:** Supabase pgvector, Gemini/OpenAI embeddings (reused OpenAI-compatible `/embeddings`), the existing AI suggestion layer.

**Key constraint:** one fixed embedding dimension (`EMBEDDING_DIM = 768`) → `vector(768)` column. Switching to a provider with a different native dimension means changing both and re-embedding.

## Tasks

### Task 1: Migration
- [ ] `0002_rag.sql`: `create extension if not exists vector`; `documents` (teacher-scoped); `document_chunks` (teacher_id, document_id, content, `embedding vector(768)`, metadata); RLS owner-scoped; `hnsw` cosine index; `match_document_chunks(query_embedding vector(768), match_count int)` returning teacher's chunks ordered by cosine distance (`auth.uid()` filtered).

### Task 2: Embeddings layer
- [ ] `lib/ai/embeddings/types.ts`: `EMBEDDING_DIM = 768`, `EmbeddingProvider { name; embed(texts): Promise<number[][]> }`
- [ ] `openai-compatible-embed.ts`: POST `{baseUrl}/embeddings`, return ordered vectors
- [ ] `providers/gemini.ts` (default, `text-embedding-004`, native 768), `providers/openai.ts` (`text-embedding-3-small`, `dimensions: 768`)
- [ ] `index.ts`: `getEmbeddingProvider()` (env `EMBEDDING_PROVIDER`, default `gemini`), `isEmbeddingConfigured()`

### Task 3: Chunking + data access
- [ ] `lib/chunk.ts`: `chunkText(text)` → ~800-char chunks with overlap
- [ ] `lib/types.ts`: add `Document`
- [ ] `lib/documents.ts`: `listDocuments()`, `matchChunks(embedding, k)` (via `rpc`)

### Task 4: Actions + RAG wiring
- [ ] `lib/actions.ts`: `createDocument(name, content)` (chunk → embed → insert doc + chunks), `deleteDocument(id)`
- [ ] `lib/ai/context.ts`: accept optional `references: string[]`, add a "your material" section + system guidance to prefer it
- [ ] `generateSuggestion`: if embeddings configured, embed a query from the student's situation, `matchChunks`, pass as references (wrapped in try/catch — never block generation)

### Task 5: UI
- [ ] `components/document-form.tsx`, `components/document-list.tsx`
- [ ] `app/(app)/materials/page.tsx` (list + upload + empty state)
- [ ] `components/nav-links.tsx`: add Materials

### Task 6: Config + build
- [ ] `.env.local.example` + `README.md` (Phase 3 + embeddings)
- [ ] `npm run build` passes

## Verification notes

- `npm run build` must pass. RAG requires running `0002_rag.sql`, an embeddings key (Gemini free tier works), and uploaded material. Retrieval + grounded generation can't be exercised here without those. Without an embeddings key, materials upload shows a hint and suggestions still generate ungrounded.
